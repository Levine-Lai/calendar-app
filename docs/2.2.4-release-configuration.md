# 2.2.4 发布与配置清单

这份清单说明“观赛日记”赛程、蓝鸟新闻自动同步和手机通知需要哪些配置。当前新闻正文为 MLB 英文原文，不使用 DeepSeek 翻译。

## 1. 当前已完成的配置

- Android 包名：`com.local.sportscalendar`。
- Firebase Android 配置：`android/app/google-services.json`。
- GitHub Actions 工作流：`.github/workflows/blue-jays-news.yml`。
- GitHub Actions 仓库权限：`Read and write permissions`。
- GitHub Secret：`FIREBASE_SERVICE_ACCOUNT_JSON`，内容为 Firebase Admin 服务账号 JSON。
- FCM 主题：`toronto_blue_jays_news_en`。
- 新闻静态数据：`public/news/blue-jays.json`。
- 新闻网络地址：jsDelivr 与 GitHub Raw 双线路，Android 原生请求作为并行后备。
- Firebase 服务账号、OAuth、FCM 权限和消息格式已通过线上 `validate_only` 校验。

## 2. 手机首次安装后的配置

1. 安装 `releases/sports-calendar-2.2.4-debug.apk`。同一 Debug 签名允许直接覆盖旧版并保留 App 数据。
2. 打开 App 左侧栏，在“多伦多蓝鸟新闻”中开启“准实时推送”。
3. Android 弹出通知授权时选择允许。
4. vivo 设置中允许“观赛日记”自启动、后台运行和后台联网。
5. vivo 电池管理中将“观赛日记”设为“不限制”或高耗电后台允许。
6. 保持 Google Play 服务可用；FCM 在缺少 Google 移动服务的设备上可能无法投递。

## 3. 自动运行机制

- GitHub Actions 每 15 分钟读取一次 MLB 蓝鸟官方 RSS，并集中更新最近 20 篇英文正文。
- App 每次启动都会静默同步；保持前台时每 5 分钟同步一次，从后台恢复超过 2 分钟或网络恢复时也会同步。
- 新文章出现时，GitHub Actions 使用 Firebase Cloud Messaging 向已订阅设备发送通知，不需要手动点击刷新。
- 整体属于准实时，正常约在新闻发布后 5 至 20 分钟出现；GitHub 调度和 vivo 省电策略可能产生额外延迟。

## 4. GitHub 需要长期保留的设置

1. 仓库 `Settings -> Actions -> General -> Workflow permissions` 保持 `Read and write permissions`。
2. 仓库 `Settings -> Secrets and variables -> Actions` 保留 `FIREBASE_SERVICE_ACCOUNT_JSON`。
3. 不要把 Firebase Admin JSON、`private_key` 或完整 Secret 提交到仓库或发送到聊天。
4. 如果撤销或重新生成 Firebase 私钥，必须同步更新 GitHub Secret。
5. 公开仓库长期无活动时，GitHub 可能暂停定时工作流；届时在 Actions 页面重新启用。

## 5. Firebase 需要长期保留的设置

1. Firebase 项目中保留包名为 `com.local.sportscalendar` 的 Android App。
2. 保持 Firebase Cloud Messaging API 可用。
3. `google-services.json` 只供 Android 客户端连接 Firebase，位于 Android 工程中。
4. Firebase Admin 服务账号 JSON 只放在 GitHub Secret 中，用于服务器发送通知。
5. Firestore、Cloud Functions、Blaze 套餐和 DeepSeek API 当前均不需要；已经创建的 Firestore 可以保留但不会被读取。

## 6. 应用内“检查更新”还需要的配置

当前 APK 已生成，但还没有公开下载地址。要让旧版手机直接检测和下载 2.2.4，需要：

1. 将 APK 上传到 GitHub Release、对象存储或其他稳定的公开 HTTPS 地址。
2. 更新 `public/version.json`：`versionCode` 设为 `26`，`versionName` 设为 `2.2.4`，`apkUrl` 填写真实 HTTPS 下载地址，并填写更新说明。
3. 提交并推送更新清单，确认两个远程清单地址均可访问。
4. 在旧版 App 中点击“检查更新”，验证版本提示、下载和覆盖安装。

在 APK 没有公开地址前，不应提前修改远程清单，否则手机会看到无法下载的更新。

## 7. 正式签名发布的可选配置

当前交付的是 Debug APK，适合个人安装测试。如果以后发布应用市场或长期公开分发，应创建并永久备份正式 keystore，然后设置：

- `SPORTS_CALENDAR_KEYSTORE`
- `SPORTS_CALENDAR_STORE_PASSWORD`
- `SPORTS_CALENDAR_KEY_ALIAS`
- `SPORTS_CALENDAR_KEY_PASSWORD`

配置后运行 `npm run build:android:release`。正式版首次发布后必须一直使用同一个 keystore，否则 Android 无法覆盖升级。
